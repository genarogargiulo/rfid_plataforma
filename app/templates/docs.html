Introducción

Visión general

Plataforma de trazabilidad de activos mediante tecnología RFID. Desarrollada por TECSIDEL.

La plataforma es una aplicación web multi-reader que conecta simultáneamente a N lectores RFID, captura las lecturas de los tags que cruzan los portones o puntos de control instrumentados, las persiste en SQL Server, y las expone en tiempo real a través de un panel web y de una API REST.

Componentes principales

  1. Backend Python (Flask) — servidor web, API REST, WebSocket, orquestación de hilos
  2. Módulo RFID (sllurp) — conexiones LLRP a los lectores físicos
  3. Capa de datos (pyodbc) — acceso a SQL Server con buffer de escritura
  4. Frontend web — panel en vivo, informes estadísticos, configuración
  5. Base de datos SQL Server — persistencia completa de lecturas, configuración y logs

Inicio rápido

Inicio rápido

Pasos para poner en marcha el sistema desde cero.

1. Requisitos previos

  • Python 3.9 o superior (python --version)
  • Microsoft SQL Server (Express o superior)
  • ODBC Driver 17 o 18 for SQL Server instalado en el servidor donde correrá la app
  • Acceso de red entre el servidor de la app y cada reader RFID (puerto TCP 5084)

2. Instalar dependencias Python

cd rfid_plataforma
python -m venv venv
venv\Scripts\activate        # Windows
pip install -r requirements.txt

Contenido de requirements.txt:

Flask==3.1.3
Flask-SocketIO==5.6.1
python-socketio==5.12.0
sllurp==3.0.5
eventlet==0.41.0
pyodbc==5.3.0

3. Crear la base de datos

-- En SQL Server Management Studio (SSMS):
CREATE DATABASE RFID_FPT;

-- Luego ejecutar sql/schema.sql contra esa base

4. Configurar conexión

Editar app/core/config.py:

DB_DRIVER  = "ODBC Driver 17 for SQL Server"
DB_SERVER  = "localhost\\SQLEXPRESS"   # o "GEN01\\SQL"
DB_DATABASE = "RFID_FPT"
DB_TRUSTED_CONNECTION = True           # Windows Auth

5. Verificar conexión

cd app
python diagnostico_sql.py

6. Arrancar la plataforma

python app.py

Abrir http://localhost:5000 → Configuración → agregar el primer reader y sus antenas. La conexión LLRP al reader se establece automáticamente en segundos.

Nota La documentación técnica está disponible en http://localhost:5000/docs (esta misma página). No existe enlace desde ninguna otra pantalla de la aplicación.

Modelo de concurrencia

La aplicación es multi-hilo. Flask-SocketIO corre con async_mode="threading", lo que permite que cada reader opere en su propio hilo de forma completamente independiente, sin interferir con el servidor HTTP ni con los demás readers.

Nombre del hiloResponsabilidadCiclo de vida
MainThreadServidor Flask/SocketIO — atiende requests HTTP y conexiones WebSocketToda la vida del proceso
reader-{id}Conexión LLRP a un reader físico. Un hilo por reader activo. Reconecta automáticamente con backoff de 5 sDesde alta del reader hasta su baja o cierre de app
recarga-config-readersConsulta la BD periódicamente para detectar altas/bajas de readers sin reiniciar la appDesde GestorMultiReader.iniciar()
buffer-lecturas-flushVuelca el buffer de lecturas en memoria a SQL Server cada N segundosDesde BufferLecturas.iniciar()
heartbeatImprime en log el estado general del sistema cada 15 s (lecturas totales, pendientes en buffer, readers activos)Desde iniciar_plataforma()

Esquema de tablas

Todas las tablas usan el esquema dbo. Script completo en sql/schema.sql.

Política de borrado lógico Ninguna tabla usa DELETE físico. Los readers y antenas se desactivan con Activo = 0 / Activa = 0. Las lecturas y logs son append-only y nunca se eliminan automáticamente.

Base de datos

Vistas analíticas

Tres vistas predefinidas para consumo directo desde Power BI, SSRS u otras herramientas sin necesidad de escribir JOINs.

dbo.vw_Lecturas

Vista desnormalizada — una fila por lectura con todos los datos de antena y reader resueltos. Punto de partida recomendado para Power BI.

SELECT
    L.LecturaId, L.EpcHex, L.EpcAscii, L.Rssi, L.TagSeenCount,
    L.TimestampReader, L.TimestampInsercion,
    A.AntenaId, A.Nombre AS NombreAntena, A.PuertoFisico, A.Ubicacion AS UbicacionAntena,
    R.ReaderId, R.Nombre AS NombreReader, R.IpAddress, R.Ubicacion AS UbicacionReader
FROM dbo.LecturasRFID L
JOIN dbo.Antenas A ON A.AntenaId = L.AntenaId
JOIN dbo.Readers R ON R.ReaderId = A.ReaderId

dbo.vw_ResumenDiarioPorAntena

Lecturas y tags únicos agrupados por día y antena. Ideal para gráficos de tendencia diaria en dashboards gerenciales.

SELECT
    CAST(L.TimestampReader AS DATE) AS Fecha,
    A.AntenaId, A.Nombre AS NombreAntena, R.Nombre AS NombreReader,
    COUNT(*) AS TotalLecturas,
    COUNT(DISTINCT L.EpcHex) AS TagsUnicos
FROM dbo.LecturasRFID L
JOIN dbo.Antenas A ON A.AntenaId = L.AntenaId
JOIN dbo.Readers R ON R.ReaderId = A.ReaderId
GROUP BY CAST(L.TimestampReader AS DATE), A.AntenaId, A.Nombre, R.Nombre

dbo.vw_UltimaLecturaPorTag

Última vez que cada tag fue detectado por cualquier antena. Útil para saber "dónde fue visto por última vez" un activo.

SELECT EpcHex, MAX(TimestampReader) AS UltimaVez
FROM dbo.LecturasRFID
GROUP BY EpcHex
Base de datos

Consultas de ejemplo

Consultas T-SQL listas para usar en SSMS o desde Power BI.

Lecturas por día (últimos 30 días)

SELECT CAST(TimestampReader AS DATE) AS Fecha,
       COUNT(*) AS TotalLecturas,
       COUNT(DISTINCT EpcHex) AS TagsUnicos
FROM dbo.LecturasRFID
WHERE TimestampReader >= DATEADD(DAY, -30, SYSUTCDATETIME())
GROUP BY CAST(TimestampReader AS DATE)
ORDER BY Fecha DESC;

Ranking de antenas por actividad (último mes)

SELECT A.Nombre AS Antena, R.Nombre AS Reader,
       COUNT(*) AS Lecturas, COUNT(DISTINCT L.EpcHex) AS TagsUnicos
FROM dbo.LecturasRFID L
JOIN dbo.Antenas A ON A.AntenaId = L.AntenaId
JOIN dbo.Readers R ON R.ReaderId = A.ReaderId
WHERE L.TimestampReader >= DATEADD(MONTH, -1, SYSUTCDATETIME())
GROUP BY A.Nombre, R.Nombre
ORDER BY Lecturas DESC;

Historial de un tag específico

SELECT L.TimestampReader, A.Nombre AS Antena, R.Nombre AS Reader,
       L.Rssi, L.EpcAscii
FROM dbo.LecturasRFID L
JOIN dbo.Antenas A ON A.AntenaId = L.AntenaId
JOIN dbo.Readers R ON R.ReaderId = A.ReaderId
WHERE L.EpcHex = '393039303030303030303030303030303030303031313138'
ORDER BY L.TimestampReader DESC;

Últimas lecturas por antena (una fila por antena)

SELECT A.Nombre, R.Nombre AS Reader,
       MAX(L.TimestampReader) AS UltimaLectura
FROM dbo.LecturasRFID L
JOIN dbo.Antenas A ON A.AntenaId = L.AntenaId
JOIN dbo.Readers R ON R.ReaderId = A.ReaderId
GROUP BY A.Nombre, R.Nombre
ORDER BY UltimaLectura DESC;

Log de caídas de readers (últimas 24h)

SELECT R.Nombre, L.Estado, L.Detalle, L.Timestamp
FROM dbo.LogEstadoReaders L
JOIN dbo.Readers R ON R.ReaderId = L.ReaderId
WHERE L.Estado IN ('desconectado', 'error')
  AND L.Timestamp >= DATEADD(HOUR, -24, SYSUTCDATETIME())
ORDER BY L.Timestamp DESC;

Distribución por hora del día

SELECT DATEPART(HOUR, TimestampReader) AS Hora,
       COUNT(*) AS Lecturas
FROM dbo.LecturasRFID
WHERE TimestampReader >= DATEADD(DAY, -7, SYSUTCDATETIME())
GROUP BY DATEPART(HOUR, TimestampReader)
ORDER BY Hora;

core/database.py

Capa de acceso a datos. Incluye las funciones CRUD sobre readers/antenas, el registro de logs, y la clase BufferLecturas para inserción en batch.

BufferLecturas

Clase que desacopla la recepción de lecturas de la escritura a BD. Las lecturas se encolan en un deque thread-safe y se vuelcan en lotes periódicos usando cursor.fast_executemany = True (optimización de pyodbc para bulk insert).

Método / PropiedadDescripción
agregar(antena_id, epc_hex, epc_ascii, rssi, tag_seen_count, timestamp_reader)Encola una lectura. Si el buffer está lleno, descarta la más antigua con un warning en log.
iniciar()Arranca el hilo buffer-lecturas-flush que ejecuta _flush() periódicamente.
detener()Detiene el hilo y ejecuta un flush final para no perder lecturas al cerrar.
pendientes (property)Cantidad de lecturas actualmente en el buffer (útil para monitoreo).
ultimo_error (property)Último error de escritura a BD, o None si el último flush fue exitoso.
Resiliencia ante caída de BD Si _flush() falla, las lecturas del lote se re-encolan al frente del buffer (orden LIFO) para el próximo intento. El buffer actúa como buffer circular: al llenarse (TAMANO_MAXIMO_BUFFER) descarta las lecturas más antiguas primero.
Módulo Python

core/rfid_manager.py

Gestión de N conexiones LLRP en paralelo. Dos clases principales: ConexionReader (un reader) y GestorMultiReader (orquestador).

ConexionReader

Encapsula la conexión LLRP a un reader físico. Cada instancia vive en su propio hilo daemon (reader-{id}).

MétodoDescripción
_factory_llrp_config()Construye el LLRPReaderConfig: antenas activas, sesión Gen2, potencia TX, report_every_n_tags=1 (crítico), y el tag_content_selector que habilita AntennaID, PeakRSSI, LastSeenTimestamp y TagSeenCount.
_callback_tag_report(reader, tag_reports)Invocado por sllurp por cada RO_ACCESS_REPORT recibido. Traduce AntennaID (puerto físico) a AntenaId (BD) usando el mapa puerto_a_antena_id. Llama al callback global on_tag_report de app.py.
conectar_y_correr()Bloquea el hilo hasta desconexión. Configura LLRPReaderClient, registra callbacks, llama connect() y luego join().
iniciar_en_hilo()Lanza _loop() en un thread daemon. El loop llama a conectar_y_correr() y, ante desconexión, espera 5 s y reintenta indefinidamente.
detener()Activa _detener_flag para que el loop no reintente, y llama reader.disconnect().

GestorMultiReader

Mantiene el diccionario reader_id → ConexionReader y recarga la configuración para detectar cambios sin reiniciar la app.

MétodoDescripción

recargar_ahora()Lanza _recargar_configuracion() en un hilo nuevo inmediatamente. Llamado por las APIs de creación/edición/borrado de readers y antenas.

_firma_antenas(reader_cfg)Devuelve tuple(sorted(puertos_fisicos)) para comparar si la lista de antenas cambió sin comparar objetos completos.
estado_actual()Snapshot de todos los readers y su estado de conexión (usado por el heartbeat log).
Módulo Python

core/epc_decoder.py

Decodificación robusta del EPC crudo entregado por sllurp, soportando los distintos formatos en que puede venir según la versión del firmware del reader.

decodificar_epc(tag: dict) → (str, str | None)

Función principal. Devuelve siempre una tupla (epc_hex, epc_ascii).

CasoEntradaepc_hexepc_ascii
Tag con EPC en bytes legibles como ASCIIb'909000000000000000001118'"393039303030...""909000000000000000001118"

_intentar_decodificar_ascii() considera texto válido solo si ≥90% de los bytes son caracteres ASCII imprimibles, y recorta padding de \x00. Esto evita que EPCs binarios que caen casualmente en rango ASCII se muestren como texto basura.

Módulo Python

core/timestamp_utils.py

Conversión de timestamps LLRP a datetime con timezone UTC, compatible con pyodbc y SQL Server.

timestamp_llrp_a_datetime(tag) → datetime

Lee el campo LastSeenTimestampUTC del tag report LLRP (microsegundos desde epoch UTC, puede venir como int directo o como dict con clave Microseconds). Convierte a datetime(timezone.utc). Si el campo no está presente, devuelve datetime.now(UTC) como fallback.

Se usa TimestampReader (del reader) en lugar de la hora del servidor para máxima precisión, dado que el reader tiene su propio reloj sincronizable con NTP.

Módulo Python

app.py

Punto de entrada de la aplicación. Orquesta la inicialización de todos los subsistemas, define las rutas HTTP, maneja los eventos SocketIO, e implementa los callbacks de tag report y estado de reader.

Callbacks principales

on_tag_report(reader_cfg, antena_id, tag)
Llamado por cada tag report recibido desde cualquier reader. Decodifica EPC, extrae timestamp, encola en el buffer para BD, actualiza el estado en memoria (ESTADO), y emite evento_lectura y kpis_actualizados por WebSocket.
on_estado_reader(reader_cfg, estado, detalle)
Llamado cuando cambia el estado de un reader (conectando/conectado/desconectado/error). Actualiza ESTADO["estado_readers"], persiste en dbo.LogEstadoReaders, y emite estado_reader por WebSocket.

Estado en memoria (ESTADO)

Diccionario global protegido por threading.Lock. Se usa para responder inmediatamente al primer cliente web que se conecta, sin necesidad de consultar la BD:

ESTADO = {
    "eventos_recientes":   deque(maxlen=300),  # últimas lecturas
    "tags_unicos":         set(),              # EPCs vistos en la sesión
    "lecturas_por_antena": defaultdict(int),   # conteo por antena_id
    "lecturas_por_reader": defaultdict(int),   # conteo por reader_id
    "estado_readers":      {},                 # reader_id → estado actual
}
Atención Este estado se pierde al reiniciar la app. Es un cache en memoria para el panel en vivo, no un reemplazo de la BD.

Inicialización (iniciar_plataforma)

  1. Prueba la conexión a SQL Server y loguea el resultado.
  2. Inicia BufferLecturas (arranca el hilo de flush).
  3. Instancia y arranca GestorMultiReader (carga readers de BD e inicia sus conexiones).
  4. Lanza el hilo heartbeat.
API REST

Crear reader

POST/api/readers
CampoTipoRequeridoDescripción
nombrestring✓Nombre descriptivo
ip_addressstring✓IP del reader en la red
puertoint—Default 5084
modelostring—Texto libre
ubicacionstring—Texto libre
session_gen2int—Default 2
tag_populationint—Default 4
tx_power_dbmint|null—null = máxima potencia

Respuesta: {"ok": true, "reader_id": 1}. Dispara GestorMultiReader.recargar_ahora() para iniciar la conexión LLRP inmediatamente.

Actualizar reader

PUT/api/readers/{reader_id}

Acepta los mismos campos que POST (todos opcionales). Actualiza solo los campos enviados. También dispara recarga del gestor.

Eliminar reader

DELETE/api/readers/{reader_id}

Borrado lógico (Activo = 0). El gestor detecta la baja en la próxima recarga y desconecta el hilo LLRP correspondiente.

Manejo de errores HTTP

Todas las rutas 404 Not Found devuelven JSON en lugar de la página de error de Flask:

{"ok": false, "error": "Ruta no encontrada: /api/ruta-inexistente"}

Esto garantiza que el frontend siempre reciba JSON parseable, aunque la URL esté mal formada.

configuracion.js — patrón de event delegation

Los botones de "+ Antena", "Editar" y "Eliminar" dentro de las tarjetas de readers se generan dinámicamente con innerHTML. Para evitar el bug clásico de onclick inline con closures rotos, se usa event delegation: los listeners se registran con addEventListener sobre cada botón después de cada llamada a renderReaders():

listaContainer.querySelectorAll(".btn-add-antena").forEach(btn => {
  btn.addEventListener("click", () => abrirModalAntena(parseInt(btn.dataset.readerId)));
});

El readerId se pasa como atributo data-reader-id en el HTML generado, no interpolado en el onclick.

WebSocket / SocketIO

El panel en vivo usa Socket.IO para recibir eventos del servidor sin polling. La conexión se establece automáticamente al cargar la página.

Eventos emitidos por el servidor

EventoCuándoPayload
estado_inicialAl conectarse un cliente nuevo{"kpis": {...}, "eventos_recientes": [...]} — precarga el estado actual sin esperar el próximo tag
evento_lecturaCada vez que un reader detecta un tag{"epc_hex", "epc_ascii", "reader_id", "reader_nombre", "antena_id", "antena_nombre", "rssi", "timestamp"}
kpis_actualizadosDespués de cada lecturaMismo shape que estado_inicial.kpis — contadores actualizados
estado_readerCambio de estado de cualquier reader{"reader_id", "nombre", "estado", "detalle"}

Modo de transporte

Flask-SocketIO usa async_mode="threading". Socket.IO negocia automáticamente entre WebSocket puro y long-polling HTTP. Los socketio.emit() realizados desde hilos de background (hilos de reader) funcionan correctamente con este modo.

Hilos del sistema

MainThreadServidor Flask — requests HTTP y eventos WebSocket
reader-{reader_id}Un hilo por reader activo. Loop de conexión LLRP con reintento automático cada 5 s.
recarga-config-readersConsulta BD cada INTERVALO_RECARGA_CONFIG_SEGUNDOS segundos para detectar cambios de readers/antenas.
buffer-lecturas-flushVuelca el buffer de lecturas a SQL Server en lotes periódicos.
heartbeatImprime en log el estado del sistema cada 15 s.

Todos los hilos son daemon threads (excepto MainThread), lo que significa que se terminan automáticamente al cerrar el proceso principal sin necesidad de señales explícitas.

Operación

Buffer de lecturas

El buffer desacopla la recepción de lecturas (producida en los hilos de reader, a alta frecuencia) de la escritura a BD (costosa, operación de red).

Ciclo de vida de una lectura

reader detecta tag
  → on_tag_report() en hilo reader-{id}
  → buffer_lecturas.agregar()                  ← O(1), thread-safe
  → buffer-lecturas-flush cada N segundos
  → cursor.fast_executemany() → SQL Server      ← bulk insert

Comportamiento ante caída de SQL Server

  1. El flush falla → el lote se re-encola al inicio del buffer (LIFO).
  2. Las lecturas siguen entrando mientras el buffer tenga capacidad.
  3. Si el buffer llega a TAMANO_MAXIMO_BUFFER, se descartan las lecturas más antiguas.
  4. Cuando SQL Server vuelve, el próximo flush inserta el lote acumulado.

El estado de error del último flush es visible en el panel de KPIs (indicador "Pendientes de guardar en BD") y en la propiedad buffer_lecturas.ultimo_error.

Operación

Protocolo LLRP

LLRP (Low Level Reader Protocol, ISO 24971-5) es el protocolo estándar para comunicación entre software y lectores RFID. La plataforma usa la librería sllurp v3.0.5.

Parámetros del ROSpec

Al conectar, se envía al reader un ROSpec (Reader Operation Spec) que define qué inventariar y cómo reportar:

antennas = [1, 2, ...]          # puertos físicos activos
session  = 2                    # sesión Gen2
tag_population = 4              # estimación para calibrar Q del anticollision
report_every_n_tags = 1         # ← crítico: reporte inmediato por cada tag
reset_on_connect = True         # limpia ROSpecs anteriores al conectar
reconnect = False               # el reintento lo maneja iniciar_en_hilo()

Tag Content Selector

Solo se solicitan los campos necesarios para minimizar el tráfico de red:

CampoHabilitadoPropósito
AntennaID✓Saber qué antena leyó el tag (para resolver antena_id en BD)
PeakRSSI✓Potencia de señal (diagnóstico y calidad de cobertura)
LastSeenTimestampUTC✓Timestamp preciso del reader
TagSeenCount✓Veces vista en la ráfaga actual
report_every_n_tags: por qué es crítico Con el valor por defecto de sllurp (None → N=0), el trigger de reporte LLRP queda en "al final del AISpec". En inventario continuo el AISpec nunca termina, así que el reader nunca emite reportes aunque esté detectando tags físicamente (se puede ver actividad en los LEDs del reader). Este bug fue detectado y resuelto durante el desarrollo; el valor siempre se fuerza a 1.

Troubleshooting

La app no conecta a SQL Server

  1. Correr python diagnostico_sql.py — lista los drivers ODBC instalados y muestra el mensaje de error exacto.
  2. Verificar que DB_DRIVER en config.py coincide exactamente con uno de los drivers listados.
  3. Verificar DB_SERVER: para instancias con nombre usar SERVIDOR\INSTANCIA (doble barra en el archivo Python: "GEN01\\SQL").
  4. Verificar que sql/schema.sql fue ejecutado contra la base indicada en DB_DATABASE.

Un reader no conecta (estado rojo en el panel)

  1. Verificar conectividad de red: ping {IP_DEL_READER}
  2. Verificar que el puerto 5084 está accesible: Test-NetConnection -ComputerName {IP} -Port 5084 (PowerShell)
  3. Asegurarse de que no hay otra sesión LLRP activa conectada al reader (LLRP admite una sola conexión simultánea).
  4. Correr python diagnostico_llrp.py editando la IP al inicio del archivo — conecta directamente sin Flask.

El reader conecta pero no llegan lecturas al panel

  1. Verificar en el log de consola que aparece starting inventory y luego Tag leído: epc=....
  2. Si el log de heartbeat muestra lecturas_totales=0 con el reader conectado, el problema está en LLRP.
  3. Confirmar que REPORT_EVERY_N_TAGS = 1 en config.py (no 0).
  4. Confirmar que las antenas configuradas en BD tienen el PuertoFisico correcto (1, 2, 3, 4) y están físicamente conectadas al reader.

El modal de antena se abre solo o no cierra

Causa conocida y resuelta: la clase CSS .modal-overlay tiene display: flex, que pisa el display: none implícito del atributo HTML hidden. Solución: agregar en style.css:

.modal-overlay[hidden] { display: none !important; }

Las lecturas no se guardan en la BD (pendientes en buffer)

  1. Verificar el indicador "Pendientes de guardar" en el panel — si crece continuamente hay un problema de escritura.
  2. Revisar logs de consola — el hilo buffer-lecturas-flush loguea los errores de INSERT.
  3. El buffer puede acumular hasta TAMANO_MAXIMO_BUFFER lecturas antes de descartar. Aumentarlo en config.py si se esperan cortes prolongados de BD.

Scripts de diagnóstico

diagnostico_sql.py

Prueba la conexión a SQL Server de forma aislada, sin Flask.

cd app
python diagnostico_sql.py

Muestra: driver ODBC configurado, servidor, base de datos, drivers ODBC instalados en el sistema, resultado de la conexión (SELECT @@VERSION), y lista de readers activos en la BD.

diagnostico_llrp.py

Prueba la conexión a un reader RFID de forma aislada, sin Flask ni BD.

cd app
python diagnostico_llrp.py

Editar READER_IP y ANTENAS al inicio del archivo. Conecta directamente al reader, arranca inventario, espera 40 segundos e imprime cada tag leído con su antena y RSSI. Al final reporta cuántas lecturas llegaron después del inicio del ROSpec (pre-start vs post-start), para distinguir lecturas reales de residuos de sesiones anteriores.
